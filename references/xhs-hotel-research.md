# 小红书酒店口碑验证工作流

## 目的

携程/飞猪圈出 TOP3 待选后, 用小红书**真实用户口碑**交叉验证, 降踩雷风险;
顺带从笔记里提取游玩内容, 反哺旅行攻略, 为每日行程规划打底。

> 先读 [`../../../trip-map-builder/references/xhs-research.md`](../../../trip-map-builder/references/xhs-research.md)
> ——那里有验证过的小红书调研方法论(搜索结果页路由、API 拦截、DOM 提取、筛选标准)。

## 主路径: 项目内置 scripts/xhs.py (推荐, 开箱即用)

小红书能力已 vendored 进 `vendor/xiaohongshu/`(源自 DeliciousBuding/xiaohongshu-skill, MIT),
仅内化酒店口碑验证所需核心模块。通过 `scripts/xhs.py` 调用, 内置反爬(频率控制+真人延迟+验证码检测):

| 能力 | 命令 | 解决的问题 |
|------|------|-----------|
| 登录态检测 | `python scripts/xhs.py check-login` | 不用手动判断登录是否失效 |
| 扫码登录 | `python scripts/xhs.py qrcode --show` | 首次登录或失效重登 |
| 搜索(带筛选) | `python scripts/xhs.py search "{关键词}" --sort-by=最新 --limit=10` | 返回结构化JSON(id/xsec_token/标题/点赞) |
| 详情提取 | `python scripts/xhs.py feed <feed_id> <xsec_token>` | 自动带xsec_token, 从`__INITIAL_STATE__`取正文, 避开300031 |

前置(clone 后一次性):
```bash
pip install -r requirements.txt && playwright install chromium
python scripts/xhs.py qrcode --show   # 首次扫码登录, cookie 存 ~/.xiaohongshu/cookies.json
```

调用示例(⚠️ 必须 `--show` 有头模式, 见下文坑):
```bash
# 1. 确认登录(失效则 python scripts/xhs.py --show qrcode)
python scripts/xhs.py check-login
# 2. 搜酒店口碑(--show 必须, 无头会导航超时)
python scripts/xhs.py --show search "那拉提英迪格 避雷" --limit 10
# 3. 取最相关的 feed_id+xsec_token 读正文
python scripts/xhs.py --show feed <id> <token>
```

### ⚠️ 两个实战坑(2026 验证, 必读)

**坑1: 无头模式导航超时 → 必须 `--show` 有头**
- 现象: `python scripts/xhs.py search ...`(无 --show) 报 `Page.goto: Timeout 120000ms exceeded, waiting until domcontentloaded`。
- 原因: 小红书搜索页有持续请求, `domcontentloaded`/`networkidle` 永不达成; 无头模式更易触发风控延迟。
- 解决: **所有 search/feed 命令都加 `--show`**(`--show` 是全局参数, 放子命令前: `python scripts/xhs.py --show search ...`)。有头模式弹窗跑, 已实测稳定。

**坑2: feed 返回的正文在 `d['note']['desc']`, 不在顶层**
- 现象: `feed` 命令返回的 JSON, 顶层 `title`/`desc` 是空, 误以为提取失败。
- 真实结构: `{"note": {"title": ..., "desc": <正文>, "user": {"nickname": ...}, "interactInfo": {"likedCount": ...}}, "comments": {...}, "widgets": [...]}`
- 解析: 取 `d['note']['desc']`(正文)、`d['note']['title']`、`d['note']['user']['nickname']`、`d['note']['interactInfo']['likedCount']`。

## 备选路(scripts/xhs.py 不可用时)

| 优先级 | 通道 | 何时用 |
|--------|------|--------|
| 🥈 | chrome-devtools-mcp | playwright未装/验证码卡住。原生MCP, 需手动处理登录态和token |
| 🥉 | OpenCLI CDPBridge 直连 | 上两者都不可用, 开 9223 调试 Chrome |

**前置(三条路都要):** 小红书需真实登录态。2026 实测**连搜索列表都要登录**("登录后查看搜索结果"),
旧版"列表免登录"已失效。

### ⚠️ 备选路详情页抓取的坑(2026 实战验证, 重要)

用 chrome-devtools-mcp 手搓时:

- **直链 navigate 到 `/explore/{id}`**: 被 `error_code=300031`("当前笔记暂时无法浏览")拦截, 多条全 404
- **fetch 抓详情页 HTML**: 全返回"页面不见了", 正则命中的是侧边栏推荐作者简介, 不是笔记正文
- **✅ 唯一可靠方式**: 留在**搜索结果页**, `click()` 卡片的 `a.cover` 元素触发站内 SPA 跳转——
  URL 会自动带上 `?xsec_token=xxx&xsec_source=pc_search`, 带 token 的页才能读到 `#detail-title` / `#detail-desc`

实现要点(chrome-devtools-mcp):
1. `navigate_page` 到搜索结果页, `evaluate_script` 等渲染(~3s)
2. JS 里 `document.querySelectorAll('section.note-item')` 取卡片, 按标题匹配目标
3. `target.querySelector('a.cover').click()` —— 点 cover, 不要点会直链跳转的元素
4. 点击会触发导航(context destroyed 属正常), 再 `evaluate_script` 等 ~3.5s 读 `#detail-desc`
5. 个别笔记仍会 `300031`(被删/限流), 跳过换下一条——但**标题本身已是有效信号**, 不必强求正文

## 搜索词模板(酒店口碑专属)

围绕"酒店名 + 城市"搜, 不搜泛词:

```
"{酒店名} 避雷"               # 🥇 首选, 直奔差评(实测最易命中负面帖)
"{酒店名} {城市} 避雷"          # 同上, 加城市消歧
"{酒店名} 真实体验"
"{酒店名} 加盟 OR 翻牌 OR 改名"  # 验加盟线索(对应 trust-system)
"{酒店名} 卫生 OR 脏 OR 虫"    # 验卫生红线(对应 review-analysis)
"{城市} {区域} 酒店推荐"        # 看有没有更好替代被漏掉
```

每家 TOP3 至少跑前 2 个模板。搜索结果页抓前 15-20 条标题筛选, 详情阅读条数按下方"收敛条件"动态定(热门≥5/小众≥3), 不固定开2-3条就收。

> **实战发现(D2奎屯万达美华实证)**: 关键词用"**避雷**"比"酒店"更易命中本质性差评帖。
> 同一家万达美华, 搜"万达美华 酒店"只到中性横评; 搜"万达美华 避雷"直接命中"狠狠踩雷"(空调失效只睡3-4小时)。
> **故"避雷"模板列为首选**, 不要图省事只搜酒店名。

## 广告/推广识别(用户强调, 重点)

小红书探店软文多, 必须过滤, 否则口碑验证失效。

### 广告特征 → 丢弃

- 通篇赞美无缺点, 像品牌宣传稿
- 多账号同时段发同一酒店相似文案(水军矩阵)
- 博主主页近期集中发同一品牌/同一商圈(探店合作)
- 只有摆拍图, 无真实体验细节(没具体房间、没具体问题、没具体时间)
- 关键词堆砌:"强烈推荐""必住""XX同款""来了一定要住"且无个人体验
- 笔记带"赞助/推广/合作"标识

### 真实信号 → 保留(复用 xhs-research.md 标准)

- 店名/地址明确, 有自己体验
- 提到**具体问题**(排队时长、隔音、味道、服务响应)——这种细节软文不爱写
- 同一槽点在多条笔记重复出现(高频词验证)
- 有缺点也有优点, 态度平衡

### 判定规则

```
单条笔记:
  广告特征 ≥2 条 → 标记"疑似软文", 丢弃
  真实信号 ≥2 条 → 采纳
  两者都不足 → 边际, 仅作氛围参考

整体口碑(采信笔记数按酒店热度动态调整, 不一刀切):
  热门酒店(携程评价量>500): 目标采信 ≥5 条真实笔记, 信号一致(正/负面方向稳定)才下结论
  小众酒店(携程评价量<500): 目标采信 ≥3 条即可, 笔记本身就少
  采信笔记中差评占比 > 50% → 🔴 高风险信号
  采信笔记中好评占比 > 70% 且无具体细节 → ⚠️ 存疑(可能是软文集中)
```

### 收敛条件(读到什么程度可下结论)

- ✅ 已达目标条数(热门≥5/小众≥3)且**信号方向一致** → 可下结论
- ⚠️ 达条数但信号矛盾(如2正3负) → 再读3条, 直到方向稳定
- ❌ 真实笔记不足(热门<5/小众<3 且无更多) → 标注"小红书样本少, 仅参考", 不下强结论, 以携程差评为主判据

## 交叉验证输出格式

接到差评分析后面, 作为独立维度。**小红书口碑与携程差评打架时, 以具体细节多的一方为准。**

```markdown
**📖 小红书口碑验证**:
- 采信笔记: X 条(丢弃软文 Y 条)
- 口碑倾向: 🟢正面 / 🟡褒贬不一 / 🔴负面
- 与携程差评一致性: ✅一致 / ⚠️有出入 / ❌打架
- 关键发现: "[小红书独有的信息, 携程没说的]"
- 风险调整: [踩雷风险是否升降级及理由]
```

示例(打架场景):
```markdown
**📖 小红书口碑验证**:
- 采信笔记: 5 条(丢弃软文 3 条)
- 口碑倾向: 🟡褒贬不一
- 与携程差评一致性: ❌打架
- 关键发现: "携程差评集中在装修味(成长性), 但小红书有2条提到'加盟后服务下滑',
  印证加盟翻牌嫌疑——携程评分4.6可能是历史高分拖累"
- 风险调整: 踩雷风险 🟢低 → 🟡中(建议确认直营)
```

## 反哺攻略(顺带, 不额外搜)

读笔记时, 除了口碑, 顺手提取**与酒店无关的游玩信息**:

- 目的地小众机位、出片角度
- 周边/景区隐藏玩法、避坑(排队时机、踩雷餐厅)
- 亲子项目真实体验(骑马、温泉、篝火的实际情况)

写回行程攻略:

```markdown
## 📌 小红书反哺(来自{酒店名}口碑调研)
- 游玩: {小众机位/隐藏玩法}
- 避坑: {排队时机/踩雷点}
- 亲子: {项目真实体验}
```

这部分喂给后续的每日行程规划(trip-map-builder Phase 1)做底料。

## 边界

- 小红书是**交叉验证 + 补充**, 不是主判据。主判据仍是携程评分/差评 + 品牌信任。
- 样本不足(热门<5/小众<3 条真实笔记)时不放大结论, 以携程差评为主判据。
- 软文识别靠特征匹配, 会有误判; 遇到"全平台一致好评"反而要警惕水军。
