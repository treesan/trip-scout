## 房型筛选规则（硬性）

### 默认房型推断(按出行人员)

首次搜索前先问(若 MEMORY.md 已记录出行模式则 smart skip)出行人员构成,据此推断默认房型:

| 出行类型 | 默认达标房型 | 说明                  |
|----------|---------|---------------------|
| 家庭出游(两大一小) | **双床房或亲子房** | 2×1.2m及以上双床 或 2×1.2m/1×2m特大床亲子房，**选含早实付价最低的达标房型** |
| 家庭出游(两大两小) | **家庭房** | 需双床+加童床或套房          |
| 情侣/夫妻(两大) | 大床房 | 大床≥1.8m即可           |
| 商务/单人 | 大床房或双床 | 按偏好                 |

用户可随时主动指定房型覆盖默认。**两大一小=双床房或亲子房(选低价达标房型)**是本项目默认(已记入MEMORY)。

### 比价房型统一规则(强制)

所有酒店比价必须选**达标房型+含双早**作为统一基准, 否则不可比:
- A酒店双床含早 vs B酒店大床不含早 = 无效对比
- 必须取"达标房型 + 含双早"的实付价

#### 达标房型定义（满足任一即可）

| 房型 | 达标条件 | 说明 |
|------|---------|------|
| 双床房 | 2×1.2m 单人床及以上 | 标准双床即达标 |
| 亲子房 | 2×1.2m 或 1×≥2.0m特大床 | 两种床型配置都满足两大一小需求 |
| 家庭房 | 1大床+1小床/沙发床 | 最佳选择 |
| 大床房(如无上述房型) | ≥ 2.0m 特大床 | 1.8m的大床三个人睡不下 |

⚠️ **选低价达标房型**：同一酒店可能有多种达标房型（如亲子房和豪华双床房），**必须选含双早实付价最低的达标房型**，不能默认选双床房。

> **示例**：某酒店亲子房(2×1.2m或1×2m特大床)含早¥987 < 豪华双床房(2×1.2m)含早¥1,038。两者都达标，选更便宜的亲子房。

会员送早/有券的情况, 按"携程不含双早价"折算到含早效果:
- 亚朵金卡(有早餐券) → 携程**不含双早价**即达含早效果
- 华住会企业铂金(送双早) → 携程**不含双早价×0.85**(85折+送早); **提前3天预订额外92折**(即不含早价×0.85×0.92)
- 万豪金卡(**不送早、房价不打折**) → **携程/飞猪含双早价**(必须订含早房型, 金卡不省这钱)
- IHG洲际优悦会 → **IHG会员含早价**(含早价通常比不含早仅贵¥100-200, 比携程含早挂牌价便宜)
- 携程钻石贵宾 → **携程钻石贵宾价**(部分酒店有专属会员价, 需登录查看)
- 旅悦会员 → **旅悦会员价**(适用于柏纳等旅悦集团旗下酒店)
- **万达系(锦华/美华/文华/嘉华)无华住会权益** → 直接用携程/飞猪含早挂牌价, 不折算

### 床型最低标准（已合并至"达标房型定义"，此处保留兼容引用）

达标房型的完整定义见上方"达标房型定义"表。核心规则：**2张床(≥1.2m×2) 或 1张≥2.0m特大床** 都满足两大一小需求，选含早实付价最低的达标房型。

### 反例警示（通用教训）

> ⚠️ **品牌信任高 ≠ 具体店品质高**：同一品牌旗下不同门店品质差异巨大。品牌信任只是起点，必须对每家店单独跑差评分析+小红书验证，不能因品牌信任就跳过。
>
> ⚠️ **达标房型不只看双床房**：同一酒店可能有多款达标房型（双床房、亲子房、家庭房），**必须选含早实付价最低的达标房型**，不能默认只看双床房。亲子房/家庭房可能比双床房更便宜且同样达标。
>
> ⚠️ **先跑五星基准再定四星**：不应因四星价格便宜就跳过五星。五星可能仅贵一点但品质高一个档次，必须先跑五星基准对比后再决定四星是否"性价比高"。
>
> ⚠️ **万达系 ≠ 华住系**：万达系酒店(锦华/美华/文华/嘉华)是万达集团独立运营，**不属于华住集团，没有任何华住会权益**。比价时直接用携程/飞猪含早挂牌价，不可折算华住折扣。

### 房型信息获取

搜索时需要额外查询房型详情：
- 飞猪: `flyai search-hotel` 返回结果包含房型信息
- 携程: 问道查询时明确要求"双床房含双早"

查询模板调整：
- "搜索{目的地}{日期}的{星级}酒店，要**达标房型含双早**（双床房或亲子房），{其他需求}"
- 搜索时需关注所有达标房型（双床/亲子/家庭），**不局限于双床房**

### 会员价格考虑

搜索时需考虑用户会员权益对实际价格的影响：

| 平台/品牌 | 会员等级 | 房价折扣 | 早餐 | 其他权益 |
|-----------|----------|---------|------|---------|
| 携程 | 钻石贵宾 | 部分酒店钻石贵宾价(需登录查看) | — | — |
| 万豪(飞猪) | 金卡 | **不打折** | **不送** | 大概率升房一级, 餐饮85折, 延迟退房 |
| 飞猪 | F3/F4 | — | — | 直通万豪金卡/希尔顿金卡 |
| 亚朵 | 金卡 | 会员价 | 有早餐券(等同含早) | 延迟退房 |
| 华住会 | 企业铂金 | 85折; **提前3天预订额外92折** | **送双早** | 最高级别 |
| IHG洲际优悦会 | 会员 | IHG会员价(含早价优于携程) | 含早价通常比不含早仅贵¥100-200 | — |
| 旅悦会员 | 会员 | 旅悦会员价 | — | 适用于柏纳等旅悦集团旗下 |
| **万达系(锦华/美华/文华/嘉华)** | **无华住权益** | **无折扣** | **无早餐券** | 万达独立运营, 不属于华住集团 |

⚠️ **万豪金卡关键认知**: 房价不打折、不送早, 所以万豪系酒店比价时**必须直接用携程/飞猪含双早挂牌价**, 不能像华住/亚朵那样折算。万豪金卡的价值在"可能升房+餐饮85折", 不在房费本身。

⚠️ **万达系关键认知**: 万达系酒店(锦华/美华/文华/嘉华)是**万达集团旗下独立运营品牌，不属于华住集团，没有任何华住会权益**。之前曾错误地将华住铂金85折+送早应用于万达系酒店(2026-07-09纠正)。万达系比价时**直接用携程/飞猪含双早挂牌价或携程钻石贵宾价**，不折算。

⚠️ **华住铂金折上折**: 华住会企业铂金除了85折+送双早外，**提前3天预订还有额外92折**（即不含早价×0.85×0.92=78.2折）。此折扣需在华住会App/小程序提前3天预订才生效。

输出时标注："会员实付价(含双早): ¥XXX"

每个推荐必须说明：
- **地理位置**: 距景区/市中心距离, 交通便利性
- **周边餐饮**: 步行范围内餐饮选择
- **酒店餐饮**: 酒店自身餐厅数量、早餐质量、特色

查询模板：
- "搜索{酒店名}的评价，重点关注周边餐饮便利性和酒店餐饮情况"

### 卫生重点关注

差评分析中，**卫生干净**是最高优先级：

| 卫生评分 | 处理 |
|----------|------|
| ≥ 4.8 | 🟢 卫生优秀 |
| 4.5-4.7 | 🟡 卫生达标 |
| < 4.5 | 🔴 卫生风险, 高度警惕 |
| 差评中"脏/不干净/有虫"占比 > 5% | 🔴 直接降级或过滤 |

卫生差评关键词：
```
脏 / 不干净 / 有虫 / 蟑螂 / 头发 / 异味(非装修)
床单不干净 / 毛巾脏 / 马桶脏 / 地板脏
```

## 双平台并行查询

### 飞猪 (flyai)

```bash
# 通用酒店搜索
flyai search-hotel \
  --dest-name "杭州" \
  --poi-name "西湖" \
  --check-in-date 2026-07-15 \
  --check-out-date 2026-07-16 \
  --hotel-stars "4,5" \
  --sort rate_desc

# 万豪系专项搜索
flyai search-marriott-hotel \
  --dest-name "杭州" \
  --check-in-date 2026-07-15 \
  --check-out-date 2026-07-16 \
  --sort price_asc
```

### 携程 (wendao)

```bash
jq -n \
  --arg token "$WENDAO_API_KEY" \
  --arg query "搜索杭州7月15日到7月16日的五星级酒店，两大一小，亲子友好" \
  '{token: $token, query: $query}' | \
  curl -s -X POST https://wendao-skill-prod.ctrip.com/skill/query \
  -H "Content-Type: application/json" \
  -d @-
```

查询模板：
- "搜索{目的地}{入住日期}到{离店日期}的{星级}酒店，{人数}，{特殊需求}"
- "搜索{目的地}附近{星级}酒店，{预算}以内，{亲子/商务/度假}"

#### 问道查指定酒店房型含早价格（实战验证 ✅ 2026-07-10）

携程问道可查询**指定酒店指定日期的房型含早价格**，返回精确价格+含早分项，并非只给区间：

```bash
jq -n \
  --arg token "$WENDAO_API_KEY" \
  --arg query "查询2026年10月1日入住10月2日退房伊宁福朋喜来登酒店双床房含双早价格" \
  '{token: $token, query: $query}' | \
  curl -s -X POST https://wendao-skill-prod.ctrip.com/skill/query \
  -H "Content-Type: application/json" -d @-
```

查询模板：
- "查询{入住日期}入住{离店日期}退房{酒店名}双床房含双早价格"
- "查询{入住日期}入住{离店日期}退房{酒店名}{房型}含早价格"

返回示例（精确到房型+含早份数）：
```
- 高级双床房（胶囊coffee+徕芬吹风机+金可儿舒睡床品）：848人民币/晚，含每日2份早餐
- 精选双床房（同上配置）：897人民币/晚，含每日2份早餐
```

**适用场景**（价格是酒店筛选最重要的一项，各Phase都可查）：
- **Phase 2 酒店联动规划**：逐家查候选酒店含早房型实付价，统一比价基准。比 ctrip_prices.py(Playwright,需登录)更轻量快速
- **Phase 1 快速可行性检查**：查目标地酒店有无合格房型及价格
- **Phase 3 行程调整后重验证**：日期变动后快速验证新日期价格（如国庆前段vs后段、避开涨价日的差额）

⚠️ wendao返回的是**携程挂牌价**，如用户有品牌会员需按"会员价格考虑"表折算到含早实付价。wendao携程价可作挂牌价基准，验证会员折扣合理性（实战：赛里木湖桔子水晶携程¥1,542 × 华住铂金85折 ≈ ¥1,311 ≈ 实际¥1,316 ✅）。

#### 会员咨询触发(首次出现该品牌时必做)

比价前必须先确认用户在该品牌的会员等级, 否则价格对比失真:

| 首次触发品牌 | 必问 | 会员价取值规则 |
|-------------|------|---------------|
| 亚朵 | "亚朵会员等级?" | 金卡有早餐券 → 取**携程不含双早价**作为实付价 |
| 华住系(桔子水晶/全季) | "华住会会员等级?" | 企业铂金=85折+送双早+提前3天92折 → **携程不含双早价×0.85×0.92** |
| 万豪系 | "万豪会员等级?" | 金卡**房价不打折不送早** → 直接用**携程/飞猪含双早价**, 不折算(价值在升房+餐饮85折) |
| IHG(洲际/假日/英迪格) | "IHG会员等级?" | IHG会员含早价优于携程 → **取IHG会员含早价** |
| **万达系(锦华/美华/文华/嘉华)** | "有万达会员/携程钻石?" | **无华住权益** → 用携程含早价或**携程钻石贵宾价** |
| 旅悦系(柏纳等) | "旅悦会员?" | 旅悦会员价 → **旅悦会员含早价** |
| 温德姆 | 对应会员等级 | 按等级折算 |

通则: 不同会员体系统一折算到**含早实付价**再比较。万豪系因金卡不送早, 直接用含早挂牌价; 亚朵/华住因送早或券, 用不含早价折算; **万达系无华住权益, 不可折算华住折扣**; IHG会员含早价可能优于携程含早挂牌价。
会员等级已知(读MEMORY.md)则 smart skip, 不重复问。

#### 问道API受限时降级

问道API会话锁定/查不到具体酒店时, 降级路径：

**Phase 1 快速酒店可行性检查阶段**：只需 wendao-skill + flyai 快速搜索，确认有无合格酒店。不需要触发差评分析引擎，不需要携程爬虫。

**Phase 2 酒店联动规划阶段**：差评分析引擎触发，需要携程差评量化数据。降级路径：
1. **脚本抓取(主)**：`python scripts/ctrip_reviews.py <hotelId> --months 12 --pages 20`
   - API模式优先(轻量快速)：从Cookie调用移动端REST API
   - 浏览器模式降级(稳定但重)：Playwright操控真实浏览器
   - ⚠️ 首次使用需先登录：`python scripts/ctrip_reviews.py login --show`（扫码登录，Cookie自动持久化，路径由脚本管理）
   - 详见 `references/review-analysis.md` 携程差评量化规则
2. **携程问道API(补充)**：问"近一年差评分类、典型问题"拿摘要, 与脚本结果交叉
3. **脚本完全不可用时的最终降级**：只用问道摘要，**必须标注"⚠️ 未量化分析(脚本不可用)"**

降级适用场景:
- 问道返回"无法在当前会话直接查询其他酒店"
- 问道会话被锁定在单一酒店
- 问道查指定酒店房型含早价正常返回精确价格（见上方"问道查指定酒店房型含早价格"），但批量查多家酒店价格时会话易锁定，需逐家查或用ctrip_prices.py批量模式
- ⚠️ **问道主动回复"无该店/查不到"但酒店实际存在** ——不能因问道说没有就跳过该酒店，必须用脚本浏览器模式搜索验证：
  ```bash
  # 第一步：用search子命令搜索酒店获取hotelId（不需要预先知道hotelId）
  python scripts/ctrip_reviews.py search "酒店名或目的地关键词"
  # 例: python scripts/ctrip_reviews.py search "伊宁绿发洲际酒店"
  # 例: python scripts/ctrip_reviews.py search "赛里木湖 酒店" --limit 5  # 模糊搜索多结果

  # 第二步：用获取到的hotelId抓取差评
  python scripts/ctrip_reviews.py <hotelId> --no-api
  ```
  search 命令是问道"无该店"时的兜底搜索方案，通过携程全局搜索框浏览器模式获取hotelId。

小红书降级同理: chrome-devtools-mcp 直连(见 references/xhs-hotel-research.md)。

#### 榜单信息获取

携程/飞猪的榜单信息通常在搜索结果中自带，如：
- "那拉提旅游风景区豪华酒店榜 No.1"
- "赛里木湖高档酒店榜 No.2"

如果搜索结果中未包含榜单信息，额外查询：
- "搜索{目的地}的携程/飞猪酒店榜单，{豪华榜/高端榜/亲子榜}"
- "搜索{酒店名}在携程的榜单排名"

#### 榜单对比规则

| 榜单排名 | 对比规则 |
|----------|----------|
| No.1 | 对比榜单 No.2、No.3、No.4（前3个） |
| No.2-3 | 对比榜单 No.1（比它靠前的） |
| No.4+ | 对比榜单 No.1-3（前3个更靠前的） |
| 未上榜 | 不做榜单对比 |

对比维度：
- 价格差异
- 评分差异
- 特色差异（位置/设施/风格）
- 选择理由：为什么推荐这个而不是榜单更靠前的

## 筛选流程

### ⓪ 高端基准(五星基准, 强制前置, 跑完再定任何四星)

每个目的地若存在高端(五星级)酒店, **必须先把当地五星酒店纳入作为参考基准**, 跑完整套方法论(双平台价+会员折算+差评分析+小红书口碑+加盟识别), 再决定四星。

> **⚠️ 星级口径说明(重要)**: 这里的"五星"**不是国家旅游星级评定委员会评定的挂牌★★★★★**(那个无公开API, 无法自动查)。本技能判定的"五星"以以下三者综合, 按可靠性排序:
> 1. **品牌定位**: 万豪/洲际/康莱德/万达文华=五星, 万枫/福朋/AC/假日=四星(品牌定位最可靠)
> 2. **平台钻级**: 飞猪`豪华型`/携程`5钻`≈五星, `高档型`/`4钻`≈四星(平台口径, 商家可标"五星标准"实非评定)
> 3. **问道描述**: "5钻级/五星标准"(最弱, 商家自标)
>
> 输出时**不可声称"国家五星级"**, 应表述为"高端/五星标准(品牌定位+平台钻级)"。若用户要查国家评定星级, 需另查文旅部/地方文旅局公示(无API, 手动)。

为什么: 四星的"性价比"只有在和五星对比后才有意义。跳过五星基准直接定四星, 可能漏掉价格相近但品质高一个档次的五星(如伊宁绿发洲际 vs 万豪万枫), 或误判某四星"性价比高"实则五星仅贵一点。

执行:
```
1. 飞猪/问道 搜当地五星(豪华型/5钻/品牌五星) + 高端四星(不预设只要四星)
2. 五星全部纳入候选池, 与四星同流程跑(价格/会员/差评/小红书/加盟)
3. 五星 vs 四星 对比表(会员实付价差 / 信任差 / 口碑差 / 房型差)
4. 决断:
   - 五星会员实付价 ≤ 四星×1.15 → 优先五星(品质溢价小)
   - 五星明显贵 → 四星可定, 但须说明"已对比五星XXX, 贵¥XXX故选四星"
   - 五星有本质性差评/加盟风险 → 四星胜出, 须说明五星被排除的原因
```

反例(典型失误): 直接定四星(如万枫¥465), 漏了同城五星(如洲际, 品质高一个档次)。四星可能仍最优, 但没经五星基准检验就定, 逻辑链不完整。

### 主筛选流程

```
搜索结果（去重合并, 含五星基准）
  │
  ├─ ① 黑名单过滤 ──→ 直接移除
  │   查 blacklist.md，匹配酒店名/品牌+城市
  │
  ├─ ② 品牌信任过滤 ──→ 低信任品牌标注⚠️
  │   查 trust-levels.md
  │
  ├─ ③ 硬性条件过滤 ──→ 价格/开业时间/评分
  │   - 价格 > 预算上限 → 移除
  │   - 开业时间 < 2020 → 标注（不自动移除）
  │   - 携程评分 < 4.5 → 移除
  │
  ├─ ④ 差评分析 ──→ 分类处理
  │   详见 references/review-analysis.md
  │
  ├─ ⑤ 加盟/直营判断 ──→ 标注类型
  │   详见 references/hotel-trust-system.md
  │
  ├─ ⑥ 小红书交叉验证 ──→ 真实口碑校准踩雷风险
  │   仅对 TOP3 待选跑, 详见 references/xhs-hotel-research.md
  │   - 搜"{酒店名} {城市} 避雷/真实体验/卫生"
  │   - 过滤软文(通篇赞美、水军矩阵、无具体细节)
  │   - 与携程差评交叉: 打架时以细节多的一方为准
  │   - 顺手提取游玩内容反哺攻略
  │
  └─ ⑦ 综合排序 ──→ 输出推荐
      评分 × 信任等级 × 质价比 × 亲子友好度
```

## 排序算法

```
综合得分 = 评分权重 × 0.25
         + 信任权重 × 0.25
         + 性价比权重 × 0.2
         + 亲子友好权重 × 0.15
         + 房型匹配权重 × 0.15

信任权重:
  🟢 高 = 1.0
  🟡 中 = 0.7
  🟠 低 = 0.4
  ⚫ 黑 = 0 (已过滤)

性价比 = 会员实付价(含早折算后)的预算利用率, 非携程挂牌价

房型匹配:
  双床房(≥1.2m×2) = 1.0
  亲子房/家庭房 = 1.0
  大床房(≥2.0m) = 0.7
  大床房(<2.0m) = 0.3 → 标注⚠️
```

### 价格相近时的决断规则

当两家酒店会员实付价相差 ≤5% 时, **不唯价格**, 按以下优先级决断:

1. 信任梯度(万豪系>亚朵直营>IHG>万达...)
2. 小红书口碑验证结论(踩雷风险低的优先)
3. 房型与窗户(双床+落地窗 > 双床+飘窗 > 双床+小窗)
4. 地理位置(距景区/餐饮近的优先)
5. 卫生评分(≥4.7优先)
6. 服务口碑(小红书+携程差评中的服务信号)

示例:
- 酒店A双床含早 ¥1139(35㎡+落地窗, 万豪系🟢) vs 酒店B行政双床 ¥1133(33㎡+飘窗, 亚朵直营🟢)
- 价差 ¥6(<5%), 两者都🟢高信任 → 酒店A在房型面积+落地窗+万豪金卡餐饮85折上更优 → 选酒店A

### 酒店全方位对比规则(强制, 2026-07-08实战经验)

当用户要求对比两家或多家酒店时（如"桔子水晶和亚朵见野哪个好"），**必须输出全方位对比表**，不可只对比差评或只对比价格。

#### 对比必须包含的4个关键维度

| 维度 | 必填内容 | 说明 |
|------|---------|------|
| **日期** | 入住日期 | 明确哪天入住，价格随日期变化 |
| **房型** | 统一房型基准(默认双床房) | 不可比不同房型，见"比价房型统一规则" |
| **价格** | 会员折算到含早实付价 | 华住×0.85+送早、亚朵金卡=不含早价、万豪=含早挂牌价 |
| **差评** | 差评数+分类+趋势+踩雷风险 | 用浏览器模式(`--no-api`)获取精确差评 |

#### 全方位对比表模板

#### 差评内容深度分析(对比时必做)

对每家酒店的差评，列出典型差评摘录表：

```markdown
#### [酒店名] 差评分析

| 日期 | 评分 | 差评内容 | 问题类型 |
|------|------|----------|----------|
| [日期] | [评分] | [差评摘录] | [设施/卫生/服务/虚假宣传] |

**核心问题**: [归纳差评集中在哪些问题]
**问题性质**: 成长性(可忽略) / 本质性(拉黑) / 系统性(加盟)
```

#### 综合决断规则

对比后必须给出明确推荐+理由，不可让用户自己选：

```
1. 价格差>20% + 低价方差评非本质性 -> 选低价方(性价比)
2. 价格差≤20% -> 按信任梯度>差评性质>趋势>房型 优先级决断
3. 低价方差评是本质性+近期恶化 -> 即使便宜也慎选，推荐高价方
4. 两家都有本质性差评 -> 都放弃，扩大搜索找第三家
```

**反例(常见错误)**:
- ❌ 只输出差评分析，不对比日期/房型/价格
- ❌ 只用API模式差评(漏90%)，不用浏览器模式
- ✅ 全方位对比表 + 差评深度分析 + 明确推荐理由

### ⚠️ 替换首选时的强制重验证(重要, 防漏洞)

当小红书验证发现首选踩雷, 要把备选提为新首选时, **新首选必须重跑完整验证**(携程差评+小红书口碑), 不能默认"另一个就安全"。

为什么: 首选踩雷后直接提备选为新首选, 但备选当时没单独跑小红书验证, 只凭横评一句"中规中矩"。这违反"首选必须跑完整套验证"的规则。

执行:
```
首选A踩雷 → 提备选B为新首选
  → B必须重跑: ①携程/飞猪评价(评分+差评关键词) ②小红书口碑("避雷"模板)
  → B验证通过(无本质性差评) → 正式定B
  → B也有问题 → 继续往下找, 或回到A降档方案
```

注: 备选在首次筛选时若已跑过完整验证(作为TOP3之一), 可直接复用结果, 不必重跑。此规则针对"未经验证就被提为首选"的情况。
